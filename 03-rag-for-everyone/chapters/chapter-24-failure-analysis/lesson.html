Failure Analysis

Failure analysis is the habit of turning bad RAG answers into fixable engineering work. It prevents teams from randomly changing prompts, chunking, retrieval, and models without knowing what broke.

Failure taxonomy

  • Corpus gap: the needed source does not exist.
  • Parsing failure: the source exists but text/table extraction broke.
  • Chunking failure: the answer-critical context was split badly.
  • Retrieval failure: gold evidence was not found or ranked too low.
  • Injection/prompt failure: good evidence arrived but the model used it poorly.
  • Policy failure: access, freshness, or safety behavior was wrong.

Root Cause First

Do not fix the first thing you see. Start from a trace: user question, route, filters, query rewrites, candidate pool, reranker output, final context, prompt, answer, citations, and evaluator result.

Debug north starClassify the failure before changing the system.

Wrong fixes create regressions. A retrieval failure needs retrieval work. A prompt failure needs prompt work.

What To Fix First

Fix in this order: missing source, access/freshness policy, parsing, chunking, retrieval recall, ranking/reranking, context injection, grounded prompt, generation style. This order avoids polishing an answer path that lacks real evidence.

Common Mistakes

M01

Prompt-tuning every failure

If the gold chunk was never retrieved, prompt tuning is theatre.

M02

No trace ids

Without traces, failure analysis becomes memory and opinion.

M03

No incident replay set

Fixed bugs return because they were never added to evals.

Project Roadmap

Build a failure triage board that converts each failed answer into root cause, owner, fix recommendation, release gate, and replay test.
