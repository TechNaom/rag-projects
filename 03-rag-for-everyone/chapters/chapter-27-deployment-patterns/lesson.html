Deployment Patterns

Deployment is where the learning project becomes an engineering product. The right pattern depends on audience, risk, latency, data privacy, and operating maturity.

Patterns to understand

  • Static demo: public learning UI, mock corpus, no secrets.
  • Local CLI: reproducible developer workflow for indexing and evals.
  • FastAPI service: clean backend boundary for retrieval, generation, and tracing.
  • Hosted frontend: user-facing product shell with auth and feedback.
  • Managed vector DB: scalable retrieval with backups, snapshots, and governance.

Choosing a Deployment Pattern

Do not deploy everything like a startup demo. A public course project can be static. An internal policy assistant needs auth, access control, traces, and rollback. A customer-facing assistant needs SLOs, abuse controls, and support workflows.

Reference Architecture

frontend -> API gateway -> auth
  -> RAG service -> retriever/vector DB
  -> model provider -> evaluator
  -> trace store -> dashboard

Project Roadmap

Build a deployment planner that recommends local, static, API, or hosted architecture based on data sensitivity, traffic, risk, and team maturity.
