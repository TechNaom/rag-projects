Access Control and Safety

Production RAG must answer only with information the user is allowed to see. Access control is not a UI feature; it is part of retrieval, context injection, prompting, logging, and audits.

Production controls

  • Role-based retrieval: filter by user, tenant, team, role, and policy.
  • Metadata enforcement: access tags must live with chunks and documents.
  • Prompt-injection safety: documents are data, never commands.
  • Audit logs: record why evidence was included or excluded.
  • Sensitive data handling: redact, refuse, or route high-risk content.

Security Guard Mental Model

Access control is like a security guard checking badges before anyone enters a room. In RAG, every chunk needs a badge: tenant, role, document classification, source status, and expiry.

Engineering Architecture

Enforce access before retrieval where possible, after retrieval before context injection, and during answer auditing. Defense in depth matters because metadata bugs happen.

user identity -> policy resolver -> metadata filter
  -> retrieval -> post-filter -> context pack
  -> grounded prompt -> answer audit -> access log

Common Mistakes

M01

Filtering only in UI

The model may still receive restricted context. Filter at retrieval and context-pack level.

M02

No audit trail

Security incidents become impossible to explain.

M03

Stale permissions

Index permissions must update when source permissions change.

Project Roadmap

Build an access-aware retriever that accepts user claims, filters allowed chunks, reports denied chunks, and produces an audit record.
